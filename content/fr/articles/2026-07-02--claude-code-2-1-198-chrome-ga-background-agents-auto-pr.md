[Claude Code v2.1.198](https://github.com/anthropics/claude-code/releases/tag/v2.1.198) est sortie le 2026-07-01T20:45:36Z, un jour après la [v2.1.197](https://github.com/anthropics/claude-code/releases/tag/v2.1.197) qui faisait de Sonnet 5 le modèle par défaut. La version rend Claude in Chrome généralement disponible, permet aux agents en arrière-plan de committer, pusher et ouvrir une pull request en brouillon lorsqu'ils terminent un travail de code dans un worktree, ajoute une skill `/dataviz` pour la conception de graphiques et de tableaux de bord, et intègre Claude Platform on AWS à la Gateway. La version s'inscrit dans une semaine d'activité de Claude Code qui a commencé avec la [v2.1.195 et son lot de fiabilité sur les agents en arrière-plan](https://github.com/anthropics/claude-code/releases/tag/v2.1.195), poursuivie avec [la v2.1.196 et son modèle par défaut d'organisation et son correctif de confiance MCP](https://github.com/anthropics/claude-code/releases/tag/v2.1.196), puis la v2.1.197 qui impose Sonnet 5 par défaut. La v2.1.198 est la plus large des quatre: c'est une version de fonctionnalités qui livre l'automatisation de bout en bout, un assistant navigateur en GA, un nouvel upstream Gateway et une montée en qualité pour le sous-agent Explore, en plus de la file habituelle de correctifs.

Le lancement de Sonnet 5 était l'événement individuel le plus important de la semaine, et [l'article sur Sonnet 5 par défaut dans Claude Code](/articles/2026-07-01--claude-sonnet-5-default-model-claude-code-1m-context) a couvert en détail le changement de modèle de la v2.1.197 ainsi que les garde-fous cyber. La v2.1.198 est la version qui construit le produit autour de Sonnet 5: l'expérience Chrome en GA, le flux d'agent en arrière-plan qui tourne pendant que le développeur est absent, la skill `/dataviz` pour les tableaux de bord que l'agent livre désormais, et l'upstream Gateway qui permet à la même session de tourner sur Claude Platform on AWS.

## Claude in Chrome passe en GA dans Claude Code

Le changement principal est la [disponibilité générale de Claude in Chrome](https://github.com/anthropics/claude-code/releases/tag/v2.1.198). Le produit était en recherche preview depuis août 2025 et verrouillé par un drapeau de fonctionnalité, une liste d'attente et les avertissements de beta-feature sur la [page Claude for Chrome](https://claude.com/claude-for-chrome). La v2.1.198 lève les verrous dans Claude Code: le même chemin d'installation qui fonctionnait pour la beta devient le chemin stable, la porte de permissions dans le produit est honorée entre les sessions CLI concurrentes (corrigeant une régression où la porte était désactivée pour une seconde session CLI), et l'isolation par groupe d'onglets par défaut est préservée. La version conserve les mêmes avertissements de beta-feature sur la [page marketing de Claude for Chrome](https://claude.com/claude-for-chrome) parce que cette page documente l'extension Chrome autonome; c'est l'intégration dans Claude Code qui passe en GA.

Pour les utilisateurs de Claude Code, le changement pratique est que l'agent qui pilote le terminal peut désormais piloter un onglet Chrome sans quitter la session. Un utilisateur peut demander à Claude Code de remplir un formulaire, de cliquer dans un flux multi-pages ou d'extraire des données structurées d'une page, et l'agent gère le côté navigateur de la même façon qu'il gère le côté terminal. Le modèle de permissions est identique à celui de la beta: isolation par groupe d'onglets par défaut, une invite de permission explicite dans le produit pour les actions sensibles, et les mêmes avertissements sur les risques du contexte navigateur. La version n'introduit pas de nouveau palier de permissions; elle retire simplement les verrous qui rendaient la fonctionnalité inaccessible par défaut.

## Les agents en arrière-plan livrent le diff

Le plus gros changement produit de la v2.1.198 est l'automatisation de bout en bout pour les agents en arrière-plan lancés depuis `claude agents`. Avant cette version, un agent en arrière-plan qui terminait un travail de code dans un worktree s'arrêtait pour demander s'il devait committer, pusher ou ouvrir une pull request. La v2.1.198 inverse le défaut: l'agent commit les changements sur la branche de son worktree, push la branche et ouvre une pull request en brouillon sur le dépôt amont. La nouvelle surface de hook `Notification` expose deux événements auxquels les outils externes peuvent s'abonner: `agent_needs_input` (émis quand l'agent bloque sur une question) et `agent_completed` (émis quand l'agent se termine, avec ou sans succès). Les événements du hook sont le point d'intégration pour les tableaux de bord, les pings de chat et les agents de suivi; le flux d'auto-PR est le point d'intégration pour le dépôt de travail du développeur.

La même version durcit aussi la surface de fiabilité des agents en arrière-plan. La v2.1.198 corrige une régression où les tâches en arrière-plan dans les panneaux de tâches web, desktop et VS Code restaient bloquées sur `Running` après leur fin ou après la reprise d'une session; un bug où les agents en arrière-plan affichaient `Reconnecting…` toutes les 52 secondes sur macOS alors que la vue agents était ouverte; et un interblocage où `claude --bg` créait silencieusement une session non-attachable lorsqu'il était combiné avec `--print` / `-p` (les drapeaux en conflit sont désormais rejetés en amont). Les équipes d'agents reçoivent un correctif par lequel un coéquipier qui meurt sur une erreur d'API signale désormais `failed` au lead, et envoyer un message à un coéquipier bloqué le réveille pour qu'il réessaie immédiatement. La version nettoie aussi une fuite d'enregistrement de worktree: les entrées verrouillées de `.git/worktrees/` provenant d'agents tués sont désormais supprimées automatiquement.

## Skill `/dataviz`, Gateway sur AWS, montée en gamme de l'agent Explore

Trois ajouts plus petits mais pratiques complètent la version. La nouvelle skill [`/dataviz`](https://github.com/anthropics/claude-code/releases/tag/v2.1.198) donne à Claude Code des conseils de conception pour les graphiques et les tableaux de bord et est livrée avec un validateur de palette de couleurs exécutable. La skill se charge à la demande lorsqu'une session est invitée à produire un graphique ou un tableau de bord et applique les mêmes règles d'accessibilité, de contraste et d'encodage des couleurs qu'Anthropic utilise en interne; le validateur accepte une palette proposée et indique si elle passe les seuils de contraste et de daltonisme encodés par la skill. La skill est additive et vit à côté des autres skills intégrées comme `/init` et `/review`.

La Gateway de Claude Code accepte désormais Claude Platform on AWS comme upstream, exposé sous l'identifiant `anthropicAws`. Les réponses model-not-found d'un upstream avancent dans la chaîne de basculement vers le suivant, donc une session qui pointe vers plusieurs providers (par exemple, Claude Platform on AWS en production et Bedrock en pré-production) bascule vers un provider fonctionnel lorsque le modèle demandé n'est pas disponible sur le premier. La version corrige aussi un dead-end où les sessions Claude Platform on AWS et Mantle tombaient sur `'Please run /login'` à l'expiration du token STS: `awsAuthRefresh` s'exécute désormais automatiquement. Le Remote Control reste désactivé lorsque `ANTHROPIC_BASE_URL` pointe vers un hôte non-Anthropic, ce qui correspond au comportement existant sous `CLAUDE_CODE_USE_BEDROCK` / `_VERTEX` / `_FOUNDRY`.

L'agent Explore intégré, qui est le sous-agent que Claude Code lance pour l'exploration de fichiers en lecture seule, hérite désormais du modèle de la session principale (plafonné à Opus) au lieu de tourner sur Haiku. Le changement est une amélioration de la qualité des sorties: une session principale Sonnet 5 qui délègue une exploration complète du code à un sous-agent Explore obtient désormais une réponse Sonnet 5 au lieu d'une réponse Haiku. Le plafond Opus est un garde-fou de coût pour qu'une session qui a choisi Opus comme modèle principal ne cascade pas silencieusement en passes Explore propulsées par Opus à chaque sous-appel. Les sous-agents et la compaction de contexte héritent aussi désormais de la configuration d'extended thinking de la session parente, donc la délégation ne perd pas le niveau d'effort choisi par la session principale.

## Correctifs plus modestes à signaler

Les notes de version de la v2.1.198 referment une longue file de petits correctifs. Les coupures réseau brèves en milieu de réponse (`ECONNRESET`) retentent désormais avec un backoff au lieu d'annuler le tour. Les requêtes excessives du classifieur en arrière-plan lorsque des processus sandbox accédaient de manière répétée au même hôte réseau sont corrigées. Le panneau `/diff` se rafraîchit désormais lorsque l'utilisateur change de branche ou commit en dehors de la session. Les tables Markdown ne débordent plus et n'enveloppent plus leur bordure droite lorsqu'elles sont rendues en mode plein écran. La commande `/desktop` n'échoue plus avec `Cannot determine working directory` après être entré puis sorti d'un worktree. La vue de progression du workflow ne perd plus les premiers agents de la liste alors que le compteur de phase reste correct dans les sessions SDK et desktop-app. Les règles conditionnelles de `.claude/rules/` se chargent désormais lorsque le fichier cible est atteint via un chemin contenant un symlink. Le mode plan autorise désormais automatiquement les appels d'outils en lecture seule lorsqu'une session démarre en mode plan. Le nom de fork par défaut de `/branch` est désormais dérivé de la première vraie invite au lieu du résumé de compaction. La précision de la coloration syntaxique dans les blocs de code, les diffs et les aperçus de fichiers s'améliore en passant à highlight.js 11. Les indications de raccourcis clavier affichent désormais opt/cmd au lieu de alt/super lorsqu'on est connecté depuis un Mac en SSH. L'UX de retry de l'API affiche la raison de l'erreur après la deuxième tentative et remplace l'astuce du spinner par un lien vers la page de statut lorsque l'API est surchargée. `/login` ouvre désormais la boîte de dialogue de connexion depuis la vue `claude agents` au lieu d'indiquer qu'elle n'est pas disponible. Les sous-agents traitent désormais les messages de l'agent qui les a lancés comme une direction de tâche normale; un message d'agent n'est toujours jamais traité comme l'approbation de l'utilisateur.

## Ce que cela change pour les utilisateurs de Claude Code

La v2.1.198 est la deuxième de deux versions consécutives de Claude Code qui font de Sonnet 5 le nouveau modèle par défaut. La v2.1.197 était le changement de modèle; la v2.1.198 est le produit qui tourne par-dessus le changement. L'expérience Chrome en GA, le flux d'auto-PR des agents en arrière-plan, la skill `/dataviz` et l'upstream Gateway `anthropicAws` sont les quatre changements qu'un utilisateur actif de Claude Code remarque un mardi matin après `claude update`. La file de fiabilité (retry sur coupure réseau, refresh automatique du STS, nettoyage des enregistrements de worktree, événements de hook pour les outils externes) est ce qu'une équipe qui fait tourner Claude Code en CI ou contre un parc de dépôts remarque au cours du mois suivant. Les deux livrent dans la même version, et la version est la plus large en termes de fonctionnalités que Claude Code ait livrée depuis la v2.1.196 et son modèle par défaut d'organisation et son correctif de confiance MCP la veille de la v2.1.197.

La version rapproche aussi Claude Code d'un flux « set it and forget it » pour les tâches qui durent plus longtemps qu'une seule session. Une équipe qui délègue un refactor à un agent en arrière-plan reçoit désormais un PR en brouillon dans sa boîte de réception au lieu d'une invite stop-to-ask. Une équipe qui veut suivre les agents en arrière-plan sur plusieurs dépôts peut brancher les nouveaux événements du hook `Notification` sur un tableau de bord. Une équipe qui fait tourner Claude Code contre un onglet Chrome dispose désormais de la version GA de la fonctionnalité au lieu d'une recherche preview. La version est la première de 2026 où les changements principaux portent sur la délégation et l'intégration plutôt que sur le modèle lui-même, et elle pose le modèle du prochain cycle de versions: le modèle est stabilisé, et le travail se concentre sur le produit autour.
