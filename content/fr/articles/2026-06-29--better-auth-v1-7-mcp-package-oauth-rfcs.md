  - question: "Qu'est-ce que la Back-Channel Logout OIDC, et pourquoi est-ce important ?"
    answer: "OIDC Back-Channel Logout 1.0 (§2.7) permet à l'OpenID Provider de notifier chaque Relying Party qui détient des tokens d'une session terminée, de serveur à serveur, en parallèle, via un Logout Token signé.   - question: "Pourquoi validAudiences a-t-il été supprimé au profit de resources ?"
    answer: "L'ancienne API `validAudiences` confondait deux choses : l'audience d'un access token émis, et la ressource que ce token est censé protéger. RFC 8707 (Resource Indicators for OAuth 2.0), RFC 9068 (JSON Web Token Profile for OAuth 2.0 Access Tokens) et RFC 9728 (OAuth 2.0 Protected Resource Metadata) séparent ces préoccupations : un token a un claim `aud`, mais la ressource qu'il protège a son propre identifiant publié comme métadonnée protected-resource. La release déplace les identifiants de ressource vers un champ `resources` sur le client, lie les clients à des ressources spécifiques via `oauthClientResource` ou le `resources` de la Dynamic Client Registration, applique la politique de ressource aux valeurs RFC 8707 à l'émission, restreint les scopes aux allowlists de ressources, utilise la plus courte TTL configurée entre les ressources, retire les noms de claims réservés RFC 9068 des claims personnalisés et émet `jti`. [Better Auth v1.7.0-rc.0](https://github.com/better-auth/better-auth/releases/tag/v1.7.0-rc.0), livrée le 26 juin 2026, est la plus grande release de l'histoire du projet : les [notes de release](https://github.com/better-auth/better-auth/releases/tag/v1.7.0-rc.0) couvrent quatre paquets (`better-auth` cœur, `@better-auth/oauth-provider`, `@better-auth/sso`, `@better-auth/scim`) et livrent un nouveau paquet (`@better-auth/mcp`). La coupe est tellement large que les mainteneurs l'ont éclatée en sections par paquet, chacune avec ses propres listes Breaking Changes, Features et Bug Fixes, et la release est précédée d'un récit de migration explicite (CLI : `npx @better-auth/cli generate` puis `npx @better-auth/cli migrate`).

Le changement principal est celui que les développeurs IA/agent remarqueront en premier : le support MCP (Model Context Protocol) est désormais son propre paquet, `@better-auth/mcp`, construit au-dessus du fournisseur OAuth. L'assistant de route `withMcpAuth` est renommé `requireMcpAuth` (il vérifie le bearer token contre la JWKS publiée et transmet les claims JWT vérifiés au gestionnaire), le client distant est renommé `createMcpResourceClient`, et les modèles de base de données sont remaniés : `oauthApplication` devient `oauthClient`, avec de nouvelles tables `oauthRefreshToken` et `oauthClientAssertion`. `@better-auth/mcp` exige désormais aussi une option `resource` explicite ; le plugin stocke cet identifiant comme ressource OAuth, publie pour lui les métadonnées protected-resource RFC 9728 et lie les access tokens émis à cette ressource. Les configurations `mcp({ loginPage, consentPage })` existantes doivent ajouter un identifiant de ressource MCP protégé, par exemple `resource : 'https://api.example.com/mcp'`.

## Ce que signifie réellement le changement de fournisseur OAuth

Le changement côté fournisseur OAuth est le plus gros bloc des notes de release, et il implémente trois RFC qu'il fallait auparavant assembler à la main par-dessus better-auth. [RFC 8707](https://datatracker.ietf.org/doc/html/rfc8707) (Resource Indicators) et [RFC 9068](https://datatracker.ietf.org/doc/html/rfc9068) (JWT Profile for OAuth 2.0 Access Tokens) remodèlent la façon dont les tokens sont émis ; [RFC 9728](https://datatracker.ietf.org/doc/html/rfc9728) (Protected Resource Metadata) standardise la façon dont un serveur de ressources s'annonce.

L'ancienne API `validAudiences` est supprimée. Les identifiants de ressource migrent vers un nouveau champ `resources` sur le client, avec liaison via `oauthClientResource` ou le `resources` de la Dynamic Client Registration. L'émission d'access tokens applique désormais la politique de ressource aux valeurs de ressource RFC 8707, restreint les scopes aux allowlists de ressources, utilise la plus courte TTL configurée entre les ressources, retire les noms de claims réservés RFC 9068 des claims personnalisés et émet `jti`. La signature JWT peut désormais honorer des épingles par ressource via `signingKeyId` et `signingAlgorithm` ; les adaptateurs JWKS exposent `getKeyById()` et `getLatestKeyByAlg()`, la table `jwks` reçoit les colonnes nullables `alg` et `crv`, et `keyPairConfigs` peut provisionner plusieurs algorithmes dans un même trousseau. La migration ajoute `oauthResource`, `oauthClientResource` et les nouvelles colonnes `jwks`.

Par-dessus cela, [OIDC Back-Channel Logout 1.0 §2.7](https://openid.net/specs/openid-connect-backchannel-1_0.html) débarque. Avant cette release, quand la session d'un utilisateur se terminait (sign-out, `/oauth2/end-session`, révocation admin, bannissement), les access tokens détenus par les relying parties restaient utilisables jusqu'à leur propre TTL. Avec la Back-Channel Logout, l'OP signe un Logout Token `logout+jwt` par client et le POSTe à chaque `backchannel_logout_uri` enregistré, avec un court timeout par RP. L'introspection de l'access token lié renvoie `{ active: false }` et `/oauth2/userinfo` le rejette avec `invalid_token`. Les refresh tokens sans `offline_access` sont révoqués en fin de session ; les refresh tokens `offline_access` sont conservés pour que l'accès API de longue durée survive à la session navigateur. La livraison passe par le gestionnaire de tâches en arrière-plan de l'hôte quand il est configuré (Vercel `waitUntil`, Cloudflare `ctx.waitUntil`) ; sans gestionnaire, elle s'exécute inline pour ne perdre aucune notification au démontage de la requête. La découverte à `/.well-known/openid-configuration` et `/.well-known/oauth-authorization-server` annonce `backchannel_logout_supported: true` et `backchannel_logout_session_supported: true` quand le plugin JWT est activé.

Le support DPoP ([#10039](https://github.com/better-auth/better-auth/pull/10039)) et l'authentification client private_key_jwt selon [RFC 7523](https://datatracker.ietf.org/doc/html/rfc7523) ([#8836](https://github.com/better-auth/better-auth/pull/8836)) complètent le tableau. Les enveloppes d'erreur RFC 6749/7009/7662/7591 sont désormais renvoyées pour les échecs de validation sur les six points d'accès OAuth (`/oauth2/token`, `/oauth2/authorize`, `/oauth2/revoke`, `/oauth2/introspect`, `/oauth2/register`, `/oauth2/end-session`). L'authentification client est désormais liée au grant émetteur, et les indicateurs de ressource RFC 8707 sont liés au grant d'autorisation (une demande pour une ressource non couverte par l'autorisation renvoie `invalid_target`).

## Validation des réponses SAML réécrite

Le flux SAML présentait trois vrais bogues en v1.6 et antérieures, tous corrigés en 1.7. La validation InResponseTo était complètement non fonctionnelle (le code lisait `extract.inResponseTo`, toujours `undefined`, au lieu du vrai chemin samlify `extract.response.inResponseTo`). L'Audience Restriction n'était jamais validée. Le SessionIndex était stocké comme objet au lieu de la chaîne interne. Chaque bogue avait une vraie conséquence en production : la validation InResponseTo initiée par SP ne s'exécutait pas, les assertions émises pour un autre service provider étaient acceptées, et la correspondance SLO SessionIndex échouait silencieusement. La release extrait les fonctions partagées `validateInResponseTo()` et `validateAudience()` dans `packages/sso/src/saml/response-validation.ts`, ce qui élimine environ 160 lignes de logique de validation dupliquée entre les deux gestionnaires ACS.

`allowIdpInitiated` est désormais `false` par défaut (le SSO IdP-initiated est désactivé par défaut, ce qui s'aligne avec le profil d'interopérabilité SAML2Int qui recommande de ne pas utiliser le SSO IdP-initiated à cause de sa sensibilité aux attaques par injection). Les certificats de signature SAML acceptent désormais un tableau pour la rotation : `idpMetadata.cert : [ currentPem, nextPem ]`, ce qui permet aux administrateurs de publier un nouveau certificat IdP à côté de l'ancien et d'effectuer la rotation sans forcer chaque session active à se réauthentifier. Le champ `callbackUrl` sur `samlConfig` est supprimé ; l'URL ACS est dérivée de `baseURL` et `providerId`. Le point d'accès /sso/saml2/callback/:providerId est remplacé par /sso/saml2/sp/acs/:providerId (GET et POST). `spMetadata` est désormais facultatif et généré automatiquement.

## Plus modestes mais réelles : isolation SCIM, durcissement Electron, réécriture generic-oauth

Le paquet SCIM namespace les identifiants de fournisseur par organisation (`scim:{organizationId}:{providerId}` ou `scim:{providerId}` pour les fournisseurs statiques au niveau app). `defaultSCIM` est remplacé par `staticProviders`. `linkExistingUsers.trustedDomains` est supprimé ; utilisez `requireExistingOrgMembership`, `shouldLinkUser`, ou la valeur explicite `true` à la place.

Le flux de connexion Electron impose désormais PKCE S256. Le PKCE simple est rejeté ; le paramètre `code_challenge_method` disparaît et chaque code d'autorisation est vérifié en hachant le verifier avec SHA-256. Le serveur ne fait plus confiance à un header `electron-origin` pour définir l'Origin de la requête ; le client Electron envoie désormais une vraie Origin (par exemple `myapp:/`), donc mettez à niveau le client et le serveur `@better-auth/electron` ensemble et assurez-vous que le schéma de votre app figure dans `trustedOrigins`. Les entrées à schéma personnalisé dans `trustedOrigins` correspondent désormais par schéma et autorité au lieu d'un préfixe de chaîne : une entrée sans hôte comme `myapp://` fait toujours confiance à tous les hôtes de ce schéma, mais une entrée avec hôte comme `myapp://callback` ne correspond qu'à cet hôte exactement, donc elle n'est plus satisfaite par `myapp://callback.attacker.tld`.

Le plugin generic OAuth est réécrit comme fournisseur social de premier ordre conforme aux RFC ([#9069](https://github.com/better-auth/better-auth/pull/9069)). `signIn.oauth2({ providerId })` est remplacé par `signIn.social({ provider })`, `oauth2.link()` par `linkSocial()`, l'URL de callback passe de `/api/auth/oauth2/callback/:id` à `/api/auth/callback/:id`, `genericOAuthClient()` est supprimé, PKCE vaut `true` par défaut, les champs de configuration `issuer` et `requireIssuerValidation` sont supprimés (la validation d'issuer est automatique via la découverte OIDC), et `mapProfileToUser` est typé comme `OAuth2UserInfo & Record<string, unknown>`. Le plugin obsolète `oidc-provider` est supprimé.

## Le plan pour la suite

Les notes de release se terminent sur le récit de migration : `npx @better-auth/cli generate` (ou `npx auth generate` / `npx auth migrate`), appliquer la migration de schéma, puis mettre à niveau. Le plan est une 1.7.0 finale dans deux à trois semaines ; traitez la v1.7.0-rc.0 comme une préversion et épinglez votre lockfile. La surface des changements cassants est suffisamment large pour qu'une mise à niveau sans tester les chemins de migration contre le nouveau schéma serait imprudente, mais l'ensemble des fonctionnalités (MCP en paquet dédié, Back-Channel Logout, RFC 8707/9068/9728, DPoP, RFC 7523, réécriture de la validation des réponses SAML) est ce qu'il fallait au projet pour être un fournisseur OAuth crédible pour les workloads d'agents IA en 2026.
