  - question: "What is OIDC Back-Channel Logout, and why does it matter?"
    answer: "OIDC Back-Channel Logout 1.0 (§2.7) lets the OpenID Provider notify every Relying Party that holds tokens for an ended session, server-to-server, in parallel, via a signed Logout Token.   - question: "Why was validAudiences removed in favor of resources?"
    answer: "The old `validAudiences` API conflated two things: the audience of an issued access token, and the resource a token is meant to protect. RFC 8707 (Resource Indicators for OAuth 2.0), RFC 9068 (JSON Web Token Profile for OAuth 2.0 Access Tokens), and RFC 9728 (OAuth 2.0 Protected Resource Metadata) split those concerns: a token has an `aud` claim, but the resource it protects has its own identifier published as protected-resource metadata. The release moves resource identifiers to a `resources` field on the client, links clients to specific resources through `oauthClientResource` or Dynamic Client Registration `resources`, applies resource policy to RFC 8707 resource values at issuance, narrows scopes to resource allowlists, uses the shortest configured TTL across resources, and strips reserved RFC 9068 claim names from custom claims while emitting `jti`. JWT signing can now honor per-resource pins via `signingKeyId` and `signingAlgorithm`. Refresh-token TTLs use the shortest applicable lifetime across resources. Deployments that relied on a per-resource `refreshTokenTtl` longer than `refreshTokenExpiresIn` will see refresh tokens expire at the provider default instead of the longer resource value. The migration adds `oauthResource`, `oauthClientResource`, and new `j wks.alg` / `jwks.crv` columns; run `npx @better-auth/cli generate` and apply before deploying."

[Better Auth v1.7.0-rc.0](https://github.com/better-auth/better-auth/releases/tag/v1.7.0-rc.0), cut on June 26, 2026, is the largest release in the project's history: the [release notes](https://github.com/better-auth/better-auth/releases/tag/v1.7.0-rc.0) span four packages (`better-auth` core, `@better-auth/oauth-provider`, `@better-auth/sso`, `@better-auth/scim`) and ship one new package (`@better-auth/mcp`). The cut is so wide that the maintainers split it into per-package sections, each with its own Breaking Changes, Features, and Bug Fixes lists, and prefixed the release with an explicit migration story (CLI: `npx @better-auth/cli generate` and `npx @better-auth/cli migrate`).

The headline change is the one AI/agent developers will notice first: MCP (Model Context Protocol) support is now its own package, `@better-auth/mcp`, built on top of the OAuth provider. The route helper `withMcpAuth` is renamed `requireMcpAuth` (it verifies the bearer token against the published JWKS and passes the verified JWT claims to your handler), the remote client is renamed `createMcpResourceClient`, and the database models are reshuffled: `oauthApplication` becomes `oauthClient`, with new `oauthRefreshToken` and `oauthClientAssertion` tables. `@better-auth/mcp` now also requires an explicit `resource` option; the plugin stores that identifier as an OAuth resource, publishes RFC 9728 protected-resource metadata for it, and binds issued access tokens to that resource. Existing `mcp({ loginPage, consentPage })` setups should add a protected MCP resource identifier like `resource: 'https://api.example.com/mcp'`.

## What the OAuth provider change actually means

The OAuth provider change is the largest single block in the release notes, and it implements three RFCs that previously had to be hand-rolled on top of better-auth. [RFC 8707](https://datatracker.ietf.org/doc/html/rfc8707) (Resource Indicators) and [RFC 9068](https://datatracker.ietf.org/doc/html/rfc9068) (JWT Profile for OAuth 2.0 Access Tokens) reshape how tokens are issued; [RFC 9728](https://datatracker.ietf.org/doc/html/rfc9728) (Protected Resource Metadata) standardizes how a resource server advertises itself.

The old `validAudiences` API is removed. Resource identifiers move to a new `resources` field on the client, with linkage through `oauthClientResource` or Dynamic Client Registration `resources`. Access-token issuance now applies resource policy to RFC 8707 resource values, narrows scopes to resource allowlists, uses the shortest configured TTL across resources, strips reserved RFC 9068 claim names from custom claims, and emits `jti`. JWT signing can now honor per-resource pins via `signingKeyId` and `signingAlgorithm`; JWKS adapters expose `getKeyById()` and `getLatestKeyByAlg()`, the `jwks` table gets nullable `alg` and `crv` columns, and `keyPairConfigs` can provision multiple algorithms in one keyring. The migration adds `oauthResource`, `oauthClientResource`, and the new `jwks` columns.

On top of that, [OIDC Back-Channel Logout 1.0 §2.7](https://openid.net/specs/openid-connect-backchannel-1_0.html) lands. Before this release, when a user's session ended (sign-out, `/oauth2/end-session`, admin revoke, ban), access tokens held by relying parties stayed usable until their own TTL. With Back-Channel Logout, the OP signs a `logout+jwt` Logout Token per client and POSTs it to each registered `backchannel_logout_uri` with a short per-RP timeout. Introspection of the bound access token returns `{ active: false }` and `/oauth2/userinfo` rejects it with `invalid_token`. Refresh tokens without `offline_access` are revoked on session end; `offline_access` refresh tokens are preserved so long-lived API access can survive the browser session. Delivery runs through the host's background task handler when one is configured (Vercel `waitUntil`, Cloudflare `ctx.waitUntil`); without a handler it completes inline so notifications are not lost on request teardown. Discovery at `/.well-known/openid-configuration` and `/.well-known/oauth-authorization-server` advertises `backchannel_logout_supported: true` and `backchannel_logout_session_supported: true` when the JWT plugin is enabled.

DPoP support ([#10039](https://github.com/better-auth/better-auth/pull/10039)) and private_key_jwt client authentication per [RFC 7523](https://datatracker.ietf.org/doc/html/rfc7523) ([#8836](https://github.com/better-auth/better-auth/pull/8836)) round out the OAuth provider story. RFC 6749/7009/7662/7591 error envelopes are now returned for validation failures across all six OAuth endpoints (`/oauth2/token`, `/oauth2/authorize`, `/oauth2/revoke`, `/oauth2/introspect`, `/oauth2/register`, `/oauth2/end-session`). Client authentication is now bound to the issuing grant, and RFC 8707 resource indicators are bound to the authorization grant (a request for a resource the authorization did not cover returns `invalid_target`).

## SAML response validation rewritten

The SAML flow had three real bugs in v1.6 and earlier, all fixed in 1.7. InResponseTo validation was completely non-functional (the code read `extract.inResponseTo`, always `undefined`, instead of samlify's actual path `extract.response.inResponseTo`). Audience Restriction was never validated. SessionIndex was stored as an object instead of the inner string. Each bug had a real production consequence: SP-initiated InResponseTo validation did not run, assertions issued for a different service provider were accepted, and SLO SessionIndex matching silently failed. The release extracts shared `validateInResponseTo()` and `validateAudience()` into `packages/sso/src/saml/response-validation.ts`, eliminating ~160 lines of duplicated validation logic between the two ACS handlers.

`allowIdpInitiated` now defaults to `false` (IdP-initiated SSO is disabled by default, aligning with the SAML2Int interoperability profile which recommends against IdP-initiated SSO due to its susceptibility to injection attacks). SAML signing certificates now accept an array for rotation: `idpMetadata.cert: [ currentPem, nextPem ]`, so administrators can publish a new IdP cert alongside the old one and complete the rotation without forcing every active session to re-authenticate. The `callbackUrl` field on `samlConfig` is removed; the ACS URL is derived from `baseURL` and `providerId`. The /sso/saml2/callback/:providerId endpoint is replaced by /sso/saml2/sp/acs/:providerId (GET and POST). `spMetadata` is now optional and auto-generated.

## Smaller but real: SCIM isolation, Electron hardening, generic OAuth rewrite

The SCIM package namespaces provider IDs by organization (`scim:{organizationId}:{providerId}` or `scim:{providerId}` for app-level static providers). `defaultSCIM` is replaced by `staticProviders`. `linkExistingUsers.trustedDomains` is removed; use `requireExistingOrgMembership`, `shouldLinkUser`, or explicit `true` instead.

The Electron sign-in flow now mandates PKCE S256. Plain PKCE is rejected; the `code_challenge_method` parameter is gone and every authorization code is verified by hashing the verifier with SHA-256. The server no longer trusts an `electron-origin` header to set the request Origin; the Electron client now sends a real Origin (e.g. `myapp:/`), so upgrade the `@better-auth/electron` client and server together and make sure your app's scheme is in `trustedOrigins`. Custom-scheme entries in `trustedOrigins` now match by scheme and authority instead of string prefix: a host-less entry like `myapp://` still trusts every host of that scheme, but a host-bearing entry like `myapp://callback` matches that host exactly, so it is no longer satisfied by `myapp://callback.attacker.tld`.

The generic OAuth plugin is rewritten as a first-class social provider with RFC compliance ([#9069](https://github.com/better-auth/better-auth/pull/9069)). `signIn.oauth2({ providerId })` is replaced by `signIn.social({ provider })`, `oauth2.link()` by `linkSocial()`, the callback URL changes from `/api/auth/oauth2/callback/:id` to `/api/auth/callback/:id`, `genericOAuthClient()` is removed, PKCE defaults to `true`, `issuer` and `requireIssuerValidation` config fields are removed (issuer validation is automatic via OIDC discovery), and `mapProfileToUser` is typed as `OAuth2UserInfo & Record<string, unknown>`. The deprecated `oidc-provider` plugin is removed.

## The plan from here

The release notes close with the migration story: `npx @better-auth/cli generate` (or `npx auth generate` / `npx auth migrate`), apply the schema migration, then upgrade. The plan is a 1.7.0 final in two to three weeks; treat v1.7.0-rc.0 as a preview and pin your lockfile. The breaking-change surface area is large enough that upgrading without testing the migration paths against the new schema would be reckless, but the feature set (MCP-as-package, Back-Channel Logout, RFC 8707/9068/9728, DPoP, RFC 7523, SAML response validation rewrite) is what the project needed to be a credible OAuth provider for AI agent workloads in 2026.
